RSPICE / VALIDATION METHODOLOGY

A number is only useful when its contract is visible.

RSpice source includes unit, integration, oracle-replay, and corpus-style testing. This page describes the discipline and its boundaries; it intentionally does not impersonate a live, current validation dashboard or a release certificate.

01 / PRINCIPLE

Unsupported, unresolved, and timed-out are outcomes.

A simulator should not turn every difficult input into a quietly plausible waveform. The useful result can be a solved record, an explicitly unsupported capability, a non-convergence condition, a timeout, or a comparison mismatch. Each state needs a visible contract and evidence.

NOT A DASHBOARDThis site does not publish static pass counts, synthetic waveforms, or a rotating list of release versions. Those artifacts become misleading the moment the underlying source revision or corpus changes.

02 / EVIDENCE LAYERS

Validate the engine at more than one scale.

01 / LOCAL

Unit & integration

Device behavior, parser rules, matrix assembly, analysis dispatch, and user-facing contracts should have targeted tests close to the implementation.

02 / HISTORY

Oracle replay

History-coupled device behavior needs retained reference material and a comparison contract that makes the numerical path reviewable.

03 / CORPUS

External decks

ngspice and Xyce corpus harnesses exercise real input families, report omission explicitly, and prevent a hung deck from stalling the suite.

The source documentation describes probe-aware tolerance handling, watchdog-supervised corpus execution, and a separation between harness discipline and conformance status. A method can be sound even when a particular revision still has visible gaps.

03 / ACCEPTANCE CONTRACTS

Match the evidence to the analysis.

CASEUSEFUL EVIDENCEDO NOT SUBSTITUTE
DC / operating pointNode and branch values, model and option identity, convergence outcome, and tolerances.A plotted point without solver and netlist provenance.
TransientProbe-aware waveform comparison, accepted time grid or locked-grid replay where timing matters, and measurement checks.A visual overlay with no sample, tolerance, or timebase contract.
AC / frequencyComplex values, sweep parameters, model identity, and an explicit comparison transform.A magnitude trace that hides phase, interpolation, or point selection.
Unsupported pathTyped refusal or quarantine entry tied to the input and capability boundary.Silently skipping an analysis and calling the deck a pass.
QUARANTINEA quarantined deck is not a pass. It should stay in the report with its reason, reproduction path, and decision owner until the behavior becomes supported or is intentionally removed.

04 / RERUN TARGETS

Run the source-level evidence against the revision in question.

Representative validation commands are documented in the source repository. Run them from the exact checkout and retain the output with the build/toolchain context when evaluating a change.

The source repository is authoritative for current command names, feature flags, fixtures, supported platforms, and CI wiring. Do not treat this marketing page as a frozen test manifest.

05 / RELEASE EVIDENCE

A future release dashboard should be generated, not hand-written.

  • Pin every result: source commit, reference corpus revision, model bundle, host, toolchain, options, and platform belong with the report.
  • Publish the exceptions: unsupported, skipped, quarantined, and performance-limited cases must stay queryable rather than being collapsed into a headline percentage.
  • Link raw artifacts: release summaries should lead to machine-readable reports, not only charts or prose.
  • Separate product claims: engine evidence, desktop UI evidence, browser evidence, and future cloud-service evidence need independent qualification gates.
