01 / CURRENT RECORD

Use the revision you are evaluating as the record.

The workspace manifest currently identifies the source as version 0.1.0. That metadata is not being presented as a finished commercial release. A specific commit, its test output, its target platform, and its documented limitations are the meaningful unit of evaluation.

NO FICTIONThere are no public v1.0.0 or v0.9.0 product releases on this site. No static release count, benchmark total, or validation percentage should be treated as current unless it is attached to a reproducible source revision and CI artifact.

04 / RELEASE DISCIPLINE

A release page belongs after release evidence exists.

  • Pin a revision: identify the exact source commit, dependencies, toolchain, model bundle, and platform target.
  • Run the gate: retain the relevant unit, integration, corpus, and host smoke-test artifacts rather than summarizing them with an untraceable badge.
  • Ship a signed artifact: publish installer provenance, checksums, support window, known limitations, and rollback path.
  • Then write notes: versioned release notes can explain user-visible changes without turning ongoing source work into marketing theater.

05 / SOURCE HISTORY

Inspect changes where they are versioned.

The repository history is the canonical public change record while RSpice remains in active development. Use it alongside the exact build and test evidence for the revision you intend to evaluate.
