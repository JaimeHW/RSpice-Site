01 / AVAILABILITY

Source is the current delivery channel.

Use source builds when you need to inspect the engine, test a deck, evaluate an interface path, or contribute to the project. There is no claim here that a current checkout has passed a release qualification for your operating system, model library, or regulated workflow.

AVAILABLE

Rust source

The public repository provides the engine, CLI, UI, Python bindings, WASM bindings, and test material.

EXPERIMENTAL

Web lab

The browser route is a constrained, client-side evaluation target rather than a hosted compute service.

NOT YET

Commercial delivery

Signed installers, entitlement, billing, cloud storage, and support terms must exist before a commercial launch claim.

02 / NATIVE BUILD

Start with the CLI, then the graphical application.

A local build is the clearest way to understand the current behavior of a specific revision. The CLI makes the execution boundary explicit; the desktop UI is the native graphical development path.

CAUTIONUse a non-confidential evaluation circuit unless you have independently verified your local environment and workflow. Development software should be validated against your own engineering requirements before it is used for a design decision.

03 / BROWSER PREVIEW

Useful for inspection, not a cloud simulation promise.

The browser preview runs a constrained WebAssembly path in the client. It is intentionally distinct from the future RSpice connected product: it does not create a project account, provision managed runners, synchronize a design, or turn a browser tab into a team workspace.

  • Use it for: quick netlist summaries and browser-accessible DC, AC, or transient evaluation within the documented WASM boundary.
  • Do not infer: large parallel campaigns, managed storage, authoritative release performance, or browser-equivalent access to every native feature.
  • Keep it local: the current public site has no active account or project-sync layer.

04 / COMMERCIAL PATH

The future product should sell durable engineering value.

The proposed commercial model is a paid native engineering seat, free review access, and separately metered managed compute when that service exists. The public pricing section is a launch hypothesis, not checkout or an offer to transact today.

SEATS

Local workflows stay local.

Commercial licensing should cover native desktop, CLI, Python, and CI work without forcing a circuit or result into a cloud service.

USAGE

Cloud cost stays visible.

Managed runners and storage should be metered separately from seats so teams can approve the cost of large jobs before they run.

05 / ACCOUNT BOUNDARY

Do not put sign-in on the static marketing site.

When RSpice adds connected projects, live collaboration, cloud execution, publishing, billing, or organization management, identity should live at a dedicated application/account boundary. The public site should remain a fast, stateless source of product information, documentation, and public links.
