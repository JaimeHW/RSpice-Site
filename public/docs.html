RSPICE / TECHNICAL SCOPE

What the source tree exposes today.

RSpice is a Rust-based analog, mixed-signal, and RF simulator in active development. This page describes the currently documented engine and interface boundaries; it is not a promise that every path is production-ready or commercially supported.

01 / RUNTIME BOUNDARY

One simulation core, deliberately different host surfaces.

The core elaborates SPICE-compatible netlists, evaluates devices, assembles modified-nodal-analysis systems, and dispatches analyses. Native builds use the parallel and SIMD-capable paths where configured; the browser target has a deliberately smaller operational envelope.

DESKTOP

EGUI / WGPU

Schematic capture, waveform inspection, analysis views, and native file-system workflows are active development surfaces.

AUTOMATION

CLI / Python

Batch simulation, formats, golden comparisons, reporting, and NumPy-oriented results support engineering workflows and CI.

BROWSER

WASM evaluation

Client-side summaries plus DC, AC, and transient runs are exposed for evaluation; this is not the planned connected product.

STATUSThe Workbench shown on the home page is a rendered GUI concept guiding the revamp. It is distinct from the browser simulation preview and is intentionally non-interactive on the marketing site.

02 / SOURCE BUILD

Build the interface you intend to evaluate.

There are no signed production installers or paid account flows on this site today. Use a current Rust toolchain and build the source path appropriate to the evaluation you are performing.

Build time, platform support, and optional model pipelines vary with the target and local toolchain. Treat a successful local build as evaluation evidence, not a production qualification.

04 / INTERFACES

Choose the host that matches the job.

  • CLI executes requested analyses, validates and inspects decks, converts result formats, compares golden references, and emits machine-readable summaries and reports for scripted work.
  • Python exposes simulation results for programmatic analysis, including transient and AC data returned to Python-oriented numerical workflows.
  • WebAssembly runs in the browser with a single-threaded evaluation model. It is useful for local experimentation, not for large managed jobs or team collaboration.
  • Desktop is the native graphical route. The GUI is currently being redesigned; the home-page Workbench mockup explains the intended interaction model rather than claiming a released final shell.

05 / VALIDATION DISCIPLINE

Failures and unsupported paths need to stay visible.

RSpice source documentation describes unit and integration tests, oracle replay for history-coupled device behavior, and corpus harnesses against ngspice and Xyce inputs. The CI evidence for a specific revision is more meaningful than a static marketing number.

Use the validation methodology for the site’s honest boundary: it explains the intended evidence chain, but it does not present a fabricated live dashboard or a substitute for release qualification.
