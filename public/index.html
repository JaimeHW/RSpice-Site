A simulation run should carry its own evidence.

A1 / INPUT GRAPH

One elaborated circuit.

Netlist parsing, expressions, subcircuits, device evaluation, matrix assembly, and analysis dispatch belong to the same simulation record.

  • SPICE-compatible decks
  • Model and include provenance
  • Inspectable generated netlist
A2 / SOLVER STATE

Failure is output.

Unsupported model physics, non-finite results, invalid cards, and timeouts should be typed, visible conditions—not quietly rounded into an answer.

  • Explicit rejection paths
  • Watchdog-aware execution
  • Machine-readable reports
A3 / EVIDENCE PACKET

Evidence travels with a run.

Measurements, reference comparisons, seeds, options, model identity, and build information make a result reviewable after the plot window has closed.

  • JUnit and TAP reporting
  • Golden comparison
  • Input-linked result summary

FEATURE SURFACE / IMPLEMENTED ENGINE CONTRACT

DEVICE → DECK → SOLVE → EVIDENCE → EXPORT
01 / SEMICONDUCTOR MODELS

MOS and bipolar families

Native and generated model paths cover common discrete, IC, and advanced compact-model workflows, with unsupported physics rejected explicitly.

  • BSIM4 v4.8, BSIM3v3.3, VDMOS, EKV3, MOS1/2/3/6, BSIM1/2
  • Generated BSIM-SOI and EKV 2.6 built-ins where sources are bundled
  • Gummel-Poon plus generated VBIC, HICUM, and MEXTRAM where sources are bundled
02 / DEVICES & INTERCONNECT

Junctions, magnetics, and lines

The device graph extends beyond transistor primitives to the components needed for power, interconnect, and board-level decks.

  • Diode, JFET/JFET2, MES/MESA/HFET families, and GaN HEMT
  • Temperature-aware R/C/L, coupled and multi-winding transformers
  • Jiles–Atherton saturable inductors; ideal, LTRA, TXL, and CPL lines
03 / SOURCES & MIXED SIGNAL

Stimulus and behavioral glue

Source and bridge models support realistic stimulus construction and mixed-domain test benches without hiding unsupported behavior.

  • PULSE, SIN, EXP, PWL, SFFM, AM, TRNOISE, and PWL file sources
  • E/F/G/H controlled and B behavioral sources; controlled switches
  • XSPICE-style analog/digital elements, tri-state drivers, and A/D–D/A bridges
04 / DECK & MODEL PIPELINE

Elaboration with provenance

RSpice resolves the circuit before solving it, preserving the cards, parameters, libraries, and model origin that produced a result.

  • .SUBCKT, .PARAM, expressions, .INCLUDE, and .LIB
  • .OPTIONS, .TEMP, engineering suffixes, and flattened elaboration
  • Verilog-A parser, semantic IR, bytecode VM, native JIT, and generated-Rust backends
05 / AUTOMATION & VALIDATION

Runs that can fail a pipeline

The CLI treats validity and comparison as contracts, not console decoration, so automated characterization can stop on a bad answer.

  • run, info, check, compare, convert, and compile-va
  • Typed exit codes, timeouts, JUnit, TAP, JSON summaries, and measurement files
  • ngspice and Xyce corpus harnesses with explicit oracles and failure tracking
06 / INTERFACES & DATA

One engine, several controlled surfaces

Native and programmatic interfaces share the same simulation core while exposing only the analyses appropriate to each target.

  • egui/wgpu desktop Workbench, CLI, Python/NumPy, and WebAssembly bindings
  • SPICE RAW, ASCII RAW, CSV, TSV, JSON, HDF5, and Touchstone output
  • Native desktop is the primary target; browser support is experimental and mobile UI remains a design target

05 / CONNECTED RSPICE

Connected work, without surrendering the local project.

The goal is not to make a browser tab mandatory. It is to let an engineering team choose connected work when the control plane is ready: versioned projects, reversible sharing, accountable compute, and readable evidence.

01 / NOW

Local project

Files, models, engine, and results stay on the engineer’s device.

02 / PLANNED

Identity boundary

Authentication belongs to the future application service, not the static marketing site.

03 / PLANNED

Project graph

Versioned files, permissions, presence, comments, and deliberate sharing.

04 / PLANNED

Compute & evidence

Queued runs, runner policy, result records, and durable review links.

ACCOUNT DECISION

Keep the public site stateless.

Do not build sign-up and login into this static marketing site. When cloud services exist, the app should own identity at a dedicated app/account surface and return users here only for public content, billing information, and documentation.

SHARING DECISION

URLs should carry a policy, not just a token.

Future result pages should distinguish public, unlisted, and private visibility; show the exact input and run provenance; and make expiration, revocation, and export intentional controls.

COMPUTE DECISION

Meter work, not seats.

Keep local simulation independent of a cloud plan. Charge separately for managed runner time and storage so teams can see the cost of a large corner, Monte Carlo, or optimization campaign before it starts.

06 / PLANNED PRICING

License the workflow. Meter the expensive work.

These are proposed launch prices, not active checkout offers. The model is designed for commercial durability: paid engineering seats, free reviewers, and separate usage for expensive managed computation.
