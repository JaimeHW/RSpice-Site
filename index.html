Analyses

Core SPICE today, RF where it is wired.

The engine covers the shipped deck-card, CLI, and API surfaces below. Advanced RF entries are labelled by access path and validation maturity, so engine-level kernels are not mistaken for deck-card analyses.

Operating point

.op

Newton with gmin and source stepping for stubborn convergence.

DC sweep

.dc

Two-source nested sweeps, temperature sweeps, and .STEP parameter families.

AC small-signal

.ac

Complex MNA magnitude and phase, with .TF gain / Zin / Zout.

Transient

.tran

Adaptive timestep with TRAP and Gear, breakpoint-accurate sources.

Input- and output-referred spot and integrated noise.

--monte-carlo

Process and mismatch draws with gauss / agauss, seeded and exactly reproducible.

Validation

Trust is a regression suite, not a slogan.

A simulator earns its place by agreeing with the simulators you already trust — then by being honest when it doesn't. RSpice is developed against checked-in ngspice-46 references, oracle fixtures, and explicit validation contracts, so a passing deck has documented evidence rather than a visual "looks close".

  • ngspice-46 parity — the regression corpus compares saved RSpice traces row-by-row against checked-in references with probe-aware tolerances.
  • Targeted oracles — history-coupled devices use replay fixtures, and RF/math kernels use focused analytic or cross-check tests where available.
  • Open corpus — the conformance decks are vendored from the ngspice test suite; under-review cases stay visible and current pass counts belong to rerun logs.
